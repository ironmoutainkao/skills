# Skill 设计评审框架

这份框架用于帮助 `skill-optimizer` 从“结构检查”升级为“设计质量评审”。

它融合了两类方法论：

1. **Anthropic / Claude Code 的实战经验**
   - skill 不只是 Markdown 文件，而是一个文件夹
   - description 是给模型看的触发规则
   - Gotchas 是高信号区域
   - 要善用渐进式披露、脚本、模板、hooks、setup/config、memory、分发等能力

2. **Google Cloud 总结的 5 种 Agent Skill 设计模式**
   - Tool Wrapper
   - Generator
   - Reviewer
   - Inversion
   - Pipeline

---

## 一、先判断目标 skill 属于哪种模式

先不要急着改写。先判断当前 skill **最主要**属于哪一种模式，必要时可标注“主模式 + 次模式”。

### 1. Tool Wrapper

特征：
- 主要价值是给模型提供某个库、产品、系统或领域的按需知识
- 常配合 `references/` 使用
- 更强调“什么时候触发”和“读哪些资料”，而不是复杂流程

适合场景：
- 文档知识库
- SDK / API / CLI 用法说明
- 内部平台知识注入

高频问题：
- description 太弱，导致触发不到
- 知识全堆在 `SKILL.md`，没有拆到 `references/`
- 写了很多常识，没有写真正的 gotchas

### 2. Generator

特征：
- 主要价值是稳定产出结构化结果
- 常配合 `assets/` 模板和 `references/` 风格指南
- 更强调输出一致性

适合场景：
- 文档生成
- 报告生成
- 模板化写作
- 结构化脚手架

高频问题：
- 输出结构不固定
- 模板应该拆到 `assets/` 却还塞在 `SKILL.md`
- 风格规则和流程规则混在一起

### 3. Reviewer

特征：
- 主要价值是评审、诊断、打分、发现问题
- 通常先判断“哪里不对”，再给建议
- 常适合配 checklist / rubric

适合场景：
- skill 审查
- 配置评审
- 代码 review
- 质量检查

高频问题：
- 只给结论，没有评价维度
- 审查标准分散，不成体系
- 容易直接跳到修改，绕过审查本身

### 4. Inversion

特征：
- 目标不是立刻生成，而是先提问、先收集信息
- 适合存在关键输入缺口的任务
- 要求明确门槛：信息不足时禁止直接产出最终结果

适合场景：
- 项目规划
- 需求澄清
- 高定制设计
- 复杂配置方案

高频问题：
- 应该先问却没问
- 缺少“未收集完信息前不得继续”的硬门槛
- 一上来就输出大而全方案

### 5. Pipeline

特征：
- 任务必须按阶段顺序执行
- 常有检查点和阶段门槛
- 强调不能跳步

适合场景：
- 多步骤工作流
- 下载 → 分析 → 生成 → 审核 → 交付
- 高风险操作流程

高频问题：
- 步骤顺序不清楚
- 缺少阶段门槛
- 用户未确认就进入下一步
- 工作流过长却没有拆分参考文件或脚本

## 二、再判断模式是否匹配

识别完模式后，继续判断：

- 当前模式是否适合这个 skill？
- 是否本该是 Generator，却写成了杂糅的大说明书？
- 是否本该是 Inversion，却没有提问门槛？
- 是否本该是 Pipeline，却没有检查点？
- 是否其实是多个模式组合，但主次关系不清？

输出时可以这样表达：

- **主模式：Pipeline；次模式：Generator**
- **问题：当前 skill 实际是 Pipeline，但没有显式阶段门槛，导致执行稳定性不足**

## 三、用 Anthropic 经验检查 skill 设计质量

### 1. description 是否真的是触发规则

检查点：
- 是否写清楚“做什么 + 什么时候用”
- 是否容易让模型判断是否应触发
- 是否写了必要的不适用场景
- 是否过于像人类简介，而不像模型路由规则

### 2. 有没有 Gotchas

检查点：
- 是否有高信号的常见坑点
- 是否记录模型最容易犯的错
- 是否提醒误触发、误判、默认行为过强等问题
- Gotchas 是不是随着使用在增长，还是根本没有

### 3. 有没有善用文件夹能力

检查点：
- 是否把 skill 当成“文件夹能力单元”，而不是单个 Markdown
- 是否合理使用：
  - `references/`
  - `scripts/`
  - `assets/`
  - 示例文件
  - 数据文件
- 当文件数量变多时，是否有根目录或子目录索引，帮助渐进式加载
- 是否存在该拆不拆、全堆在 `SKILL.md` 的问题

### 4. 是否使用渐进式披露

检查点：
- `SKILL.md` 是否只保留高层流程
- 细节是否拆到 references / templates / checklist
- 文件较多时，是否提供索引，说明“遇到什么场景读哪个文件”
- 是否避免一开始把所有细节塞给模型

### 5. 是否写了显而易见的废话

检查点：
- 是否重复模型本来就知道的常识
- 是否缺少真正对模型有增量价值的信息
- 是否把重点放在“你容易做错什么”，而不是“这是一个很厉害的 skill”

### 6. 是否缺 setup / config 思维

检查点：
- skill 是否有首次使用需要确认的配置
- 是否应该用配置文件或稳定目录保存状态
- 是否每次都让模型重复问同样的问题

### 7. 是否有 memory / logging / history 价值

检查点：
- 这个 skill 是否适合沉淀历史记录
- 是否可以通过日志或状态文件提升稳定性
- 是否需要稳定目录，而不是把数据放在 skill 本体里

### 8. 是否该脚本化

检查点：
- 是否存在高重复、可确定、适合脚本化的动作
- 是否本来该给 helper scripts，却还在用纯文字让模型手工做
- 是否能通过脚本减少上下文消耗和操作不一致

### 9. 是否考虑外部依赖与可安装性

检查点：
- 是否依赖外部 skill、CLI、服务、运行时或系统能力
- 是否给出可直接执行的安装命令、版本要求与校验命令
- 是否让 AI 能在新环境里自动安装，而不是只写“请先安装 XXX”
- 是否说明依赖缺失时的降级方案或停机条件

### 10. 是否存在 railroading 或过度设计

检查点：
- 是否把指令写得太死，导致复用性变差
- 是否为简单场景引入过多结构
- 是否为了“显得体系化”而过度设计

### 11. 是否考虑分发与复用

检查点：
- 这是一个仅个人使用的 skill，还是适合公开分发的 skill
- 如果公开分发，是否把个人私有路径 / 偏好 / 环境依赖写死了
- 仓库 README 与 skill 描述是否能帮助他人理解何时使用

### 12. 是否考虑异常与失败场景

检查点：
- 是否覆盖缺文件、缺输入、网络失败、鉴权失败、空结果、命令失败等异常情况
- 是继续重试、降级执行、停下来提醒，还是直接失败，是否写清楚
- 是否避免在异常条件下继续执行高风险步骤

### 13. 是否存在敏感信息或敏感操作

检查点：
- 是否把 API Key、Token、Cookie、账号、内部地址等敏感信息直接写进 skill
- 如果发现敏感信息，是否会提醒确认处理，而不是直接原样分发
- 是否包含删除、覆盖、部署、发送消息、联网改状态、付费调用等高副作用操作
- 这些操作是否有确认门槛、风险提示和默认禁止条件

### 14. Skill 粒度是否合适

检查点：
- 当前 skill 是否过粗，承担多个弱相关任务
- 当前 skill 是否过细，导致触发分散、边界重叠、维护成本过高
- 如果父目录名为 `skills`，是否至少查看同级 skill 的名称与必要描述，评估是否存在可合并或应避免抢触发的对象

### 15. README 与 SKILL 是否职责分离

检查点：
- 是否存在 `README.md`
- `README.md` 是否面向人说明使用场景、主要功能、风险、边界与注意事项
- `SKILL.md` 是否专注给 AI 的执行规范、工作流与 Gotchas
- 两者是否避免大段重复；README 不应替代 SKILL，SKILL 也不应承担面向人的介绍文档职责

## 四、输出审查结果时建议包含的额外判断

除了常规的高 / 中 / 低优先级问题，还建议补充下面这些结论：

### 1. 模式判断

例如：
- 主模式：Reviewer
- 次模式：Inversion
- 当前问题：模式本身合理，但缺少明确门槛与 checklist 化标准

### 2. 设计质量判断

可以快速判断：
- 主要问题在**触发语义**
- 主要问题在**工作流设计**
- 主要问题在**结构拆分**
- 主要问题在**边界 / 确认门槛**
- 主要问题在**缺少 Gotchas / scripts / templates / config**
- 主要问题在**异常处理 / 安全边界 / 依赖可安装性**
- 主要问题在**README / 索引 / 粒度设计**

### 3. 优先级建议

建议把问题优先级按下面顺序判断：

1. 先修触发失败、边界缺失、流程不可执行
2. 再修敏感信息、危险操作缺确认、异常处理缺失、依赖不可安装
3. 再修模式错位、结构臃肿、渐进式披露不足、README / 索引缺失
4. 再修 Gotchas / scripts / assets / config 缺失、粒度不当
5. 最后再做措辞、示例和可读性优化

## 轻量审查模式

当用户只是问“有没有优化空间”“帮我看看哪里能改”“先简单审一下”时，默认先走轻量审查，不必完整穷举全框架。

轻量审查只看六项：

1. 触发语义
- `description` 是否像路由规则，而不只是简介
- 是否写清“做什么 + 什么时候用”

2. 确认门槛
- 是否存在必须确认却没有卡住的步骤
- 是否明确什么回复才算开始执行

3. 输出契约
- 是否区分审查结论、优化计划、实施结果
- 是否写清范围、不改动项、额外建议

4. 结构拆分
- `SKILL.md` 是否过胖
- 细节是否已合理下沉到 `references/`

5. 安全与依赖
- 是否存在敏感信息或高副作用操作
- 是否给出外部依赖的安装 / 校验方式

6. 文档与粒度
- 是否缺 `README.md`，或 README 与 `SKILL.md` 职责混乱
- 是否明显过粗 / 过细，或与同级 skill 边界重叠

轻量审查的目标：
- 先找出 1 到 3 个最值得改的问题
- 优先修触发失败、确认缺失、敏感操作失控、输出不稳的问题
- 不因为一次轻审就默认升级成整 skill 重构

---

## 五、常见误判提醒

- 不要为了套模式而套模式
- 不要一看到 skill 很长，就默认必须拆文件
- 不要把“用户只想微调 description”误做成整 skill 重构
- 不要因为 references 存在，就默认渐进式披露已经做得好
- 不要把“有流程”误认为“有门槛”
- 不要把“有脚本”误认为“设计就合理”
- 不要把“写了 README”误认为“README 和 SKILL 已正确分工”
- 不要把“文件很多”误认为“渐进式披露做得好”；没有索引时仍可能难以加载

---

## 六、给 `skill-optimizer` 的使用建议

在审查目标 skill 时，建议流程如下：

1. 先读 `SKILL.md`
2. 判断主模式 / 次模式
3. 再决定要不要读 references / scripts / assets
4. 用本框架检查：
   - 触发语义
   - 模式匹配
   - 工作流门槛
   - 渐进式披露
   - 异常处理
   - 依赖安装与校验
   - 安全边界与敏感操作
   - README / 索引 / 粒度
   - Gotchas
   - 脚本化 / 模板化 / 配置化 / memory 化机会
5. 最后再输出优化计划，并等待用户确认
