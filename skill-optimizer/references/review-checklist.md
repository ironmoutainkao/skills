# Skill Review Checklist

按需审查，不必每次穷举所有项目。若用户已指定优化方向，先覆盖对应部分。

判定口径：
- `通过`：这一项已经清楚、稳定，可直接复用
- `存疑`：基本可用，但边界、表达或稳定性仍有风险
- `不通过`：会影响触发、正确性或执行一致性，应优先修复

## 1. Triggering

- `name` 是否只含小写字母、数字、连字符
- 目录名是否与 `name` 完全一致
- `description` 是否同时说明“做什么”和“何时使用”
- `description` 是否包含用户真实会说的关键词
- `description` 是否避免第一人称和空泛描述
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 把“适用场景”里的触发词回收到 `description`
- 去掉模糊描述，例如“帮助处理各种内容”
- 把技能边界写清楚，避免和相邻 skill 抢触发

## 2. Workflow

- 是否有明确顺序
- 是否存在必须确认却没有等待确认的步骤
- 是否说明输入、输出、落盘位置或交付格式
- 是否对失败场景给出降级策略
- 是否覆盖异常场景，例如缺文件、缺输入、命令失败、网络失败、鉴权失败、空结果
- 是否把“应该做”和“禁止做”写清楚
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 增加 checklist
- 把“先检查再执行”写成硬性门槛
- 为关键输出增加固定格式

## 3. Progressive Disclosure

- `SKILL.md` 是否只保留核心流程
- 详细规则、示例、长说明是否拆到 `references/`
- `references/` 是否都直接从 `SKILL.md` 链接
- 文件较多时，是否提供根目录或子目录索引，帮助模型按需读取
- 是否避免多层嵌套引用
- 是否避免在 `SKILL.md` 和 `references/` 重复大段内容
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 把大段示例、规则表、术语表移到 `references/`
- 在 `SKILL.md` 里只写“什么时候读哪个 reference”

## 4. Resource Strategy

- 是否存在反复手写且适合脚本化的操作
- 是否缺少稳定执行所需的模板、脚本或参考资料
- `assets/` 是否只放输出资源，不放说明文档
- 若依赖外部 skill、CLI、服务或运行时，是否给出安装与校验命令，便于 AI 自动安装
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 把重复命令封装到 `scripts/`
- 为外部依赖补安装、验证和失败排查说明

## 5. Safety and Sensitivity

- 是否包含疑似敏感信息，例如 API Key、Token、Cookie、账号、私有地址
- 若发现敏感信息，是否避免在正文中回显完整值，并提醒确认处理方式
- 是否包含删除、覆盖、部署、发送消息、联网改状态、付费调用等高副作用操作
- 高副作用操作是否有确认门槛、风险提示和默认禁止条件
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 把明文敏感信息改成占位符或配置说明
- 为敏感操作增加“执行前确认”的硬门槛

## 6. Output Contract

- 输出是否可直接给用户或下游 agent 使用
- 是否区分“审查结论”“优化计划”“最终修改结果”
- 是否说明确认回复的触发条件，例如“确认”“开始修改”
- 是否在最终汇报中列出文件和剩余风险
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 固定 plan 模板
- 固定最终汇报模板
- 把用户可选回复写清楚

## 7. Scope Control and Granularity

- 是否明确写出本次优化范围
- 是否列出不改动项，避免悄悄扩大改动面
- 是否把超出范围的问题单列为“额外建议”
- Skill 是否过粗，导致一个 skill 承担多个弱相关任务
- Skill 是否过细，导致触发分散、边界模糊、维护成本过高
- 如果当前 skill 的上层目录为 `skills`，是否检查同级 skill 是否存在明显可合并或应避免抢触发的对象
- 可判定为：通过 / 存疑 / 不通过

## 8. Documentation

- 是否存在 `README.md`
- `README.md` 是否面向人说明使用场景、主要功能、风险和边界
- `SKILL.md` 是否保持面向 AI 的执行规范，而不是与 `README.md` 大段重复
- 索引文件、README 与 `SKILL.md` 的职责是否清楚
- 可判定为：通过 / 存疑 / 不通过

常见优化：

- 缺 README 时补 README
- 把“给人看的介绍”与“给 AI 的执行规则”分开

## 9. Prioritization

默认按这个顺序排序问题：

1. 高优先级：敏感信息、危险操作缺确认、触发失败、确认缺失、工作流错误、明显冲突
2. 中优先级：异常处理缺失、依赖不可安装、结构臃肿、资源组织差、上下文浪费
3. 低优先级：README/索引补强、粒度微调、措辞打磨、示例增强、展示优化
