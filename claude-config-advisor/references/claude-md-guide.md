# `CLAUDE.md` 指南

## 作用

`CLAUDE.md` 是 Claude Code 的环境微调文档。开启对话时，Claude 会自动把它拉入上下文，用来理解项目规范、命令入口和关键约束。

它最适合承载“全局且稳定”的项目知识，而不是零散笔记。

## 放置位置

常见做法：

- 放在项目根目录，并提交到 Git
- 放在父目录，供多个仓库共享
- 放在特定子目录，供局部上下文使用
- 使用 `CLAUDE.local.md` 承载个人偏好，并加入 `.gitignore`

## 应该写什么

优先写这些高价值信息：

- 常用 Bash 命令
- 构建、测试、类型检查入口
- 核心目录和关键文件入口
- 代码风格约束
- 测试规范
- 分支、提交、rebase/merge 等协作规则
- 开发环境要求
- 项目中的反直觉行为、坑点和警告

## 不建议写什么

- 大段背景故事
- 与项目无关的通用知识
- 很容易过时的长列表
- 本应拆到 skill / command / agent 的大段操作流程

## 编写原则

### 1. 简洁优先

`CLAUDE.md` 本质上是提示词的一部分，越短越容易稳定生效。

推荐写法：

- 用标题和短列表组织内容
- 每条规则尽量一句话讲清
- 用强提示词强调硬约束，例如 `IMPORTANT`、`YOU MUST`

### 2. 聚焦全局约束

适合放这里的是“项目级规则”：

- 默认包管理器是什么
- 测试先后顺序是什么
- 哪些文件不要改
- 哪些目录是生成产物

不适合放这里的是“某一专项任务的完整操作手册”，那更适合 skill。

### 3. 保持单一事实来源

如果项目已经有独立文档，例如 `AGENTS.md`、`CONTRIBUTING.md`、架构文档，可以在 `CLAUDE.md` 中引用，而不是重复拷贝。

例如：

```markdown
@AGENTS.md
```

## 审查标准

一个合理的 `CLAUDE.md` 通常具备这些特征：

- 读起来快，几分钟内能扫完
- 全是对当前项目真正有用的信息
- 可以直接指导 Claude 执行工作
- 能明显区分共享规范和个人偏好

## 常见问题

### 问题 1：写成百科全书

表现：

- 内容特别长
- 什么都想塞进去
- 维护成本越来越高

建议：

- 把专项工作流拆到 `.claude/skills/`
- 把可复用提示拆到 `.claude/commands/`
- 只保留全局规则和入口信息

### 问题 2：只有抽象原则，没有可执行信息

表现：

- 只有“保持代码整洁”“注意规范”之类的空话
- 没有真实命令、文件入口和具体要求

建议：

- 补上命令、路径、工具、测试入口
- 对硬规则使用更明确措辞

### 问题 3：团队共享和个人偏好混在一起

表现：

- 既写团队规范，也写个人机器习惯
- 导致团队成员难以判断哪些必须遵守

建议：

- 共享内容放 `CLAUDE.md`
- 个人内容放 `CLAUDE.local.md` 或本地配置文件
