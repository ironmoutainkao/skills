# 配置审查基线

用这份基线判断现有 Claude 配置“是否合理”，并给出分级评价。

## 结论分级

### 合理

满足以下大部分条件：

- 目录和文件与实际需求匹配
- `CLAUDE.md` 简洁且可执行
- 共享配置和本地配置分层清楚
- skills / agents / commands / hooks 各自职责明确
- 没有明显重复、冲突或过度设计

### 基本合理但有改进空间

典型表现：

- 主体方向对，但结构稍混乱
- 某些目录职责重叠
- `CLAUDE.md` 偏长或偏空
- 局部缺少复用封装

### 不太合理

典型表现：

- 配置明显脱离真实需求
- `CLAUDE.md` 混杂了大量不该放的内容
- 团队规范和个人偏好严重混用
- 大量目录存在但基本没有可维护价值
- 缺少关键配置，导致 Claude 很难稳定执行项目规范

## 审查维度

### 1. 结构匹配度

看当前配置是否与项目复杂度匹配。

正向信号：

- 小项目以 `CLAUDE.md` 为主，适度补 commands 或 skills
- 团队项目按需拆分技能、命令、agent

风险信号：

- 小项目却引入大量 hooks / output styles / agents
- 复杂协作项目却只有一个超长 `CLAUDE.md`

### 2. 职责边界

看信息是否放在正确位置。

正向信号：

- 全局规范在 `CLAUDE.md`
- 专项流程在 skills
- 快速复用提示在 commands
- 自动化脚本在 hooks

风险信号：

- `CLAUDE.md` 里写完整技能说明
- commands 里塞项目长期规范
- hooks 承担需要主观判断的任务

### 3. 可维护性

看未来是否容易继续维护。

正向信号：

- 命名清楚
- 内容短且聚焦
- 重复少
- 能快速找到入口

风险信号：

- 文件过长
- 规则重复出现在多个文件
- 描述模糊，后续维护只能靠猜

### 4. 可执行性

看 Claude 是否能真正据此稳定行动。

正向信号：

- 有具体命令
- 有明确路径
- 有硬约束提示
- 有真实触发场景

风险信号：

- 只有抽象口号
- 没有关键入口
- 命令或路径过时

## 建议输出顺序

审查结果建议按这个顺序组织：

1. 先给总评结论
2. 再说做得好的地方
3. 再列主要问题
4. 最后给按优先级排序的优化建议

## 常见优化方向

### 结构过重

建议：

- 合并低价值目录
- 先回到 `CLAUDE.md + commands` 的轻量组合
- 把没有复用价值的 skills / agents 移除或收缩

### 结构过轻

建议：

- 补充最关键的 skills 或 commands
- 将重复工作流从 `CLAUDE.md` 中拆出去

### `CLAUDE.md` 失焦

建议：

- 缩短篇幅
- 删除泛泛而谈内容
- 保留命令、入口、硬规则和例外事项
