# 因素 IV：构建前先调研

**原则：在生成一行代码之前，先理解问题空间。没有例外。**

## 为什么调研不可跳过

跳过调研的 Agent 生成的代码：
- 重复已有工具函数
- 违背已有架构决策
- 破坏代码库约定
- 隔离状态能跑，集成就崩

10 分钟的调研可以节省数小时返工。调研是投资，不是成本。

## 调研阶段的标准

**调研是独立的阶段，产出是发现文档，不是代码。**

✅ 好的调研：
```markdown
# 调研发现：认证系统

## 现有实现
- 位置：src/auth/authenticator.py
- 模式：JWT + refresh token
- 相关文件：src/auth/middleware.py、tokens.py

## 关键决策（来自 ADR）
- Token 15 分钟过期（ADR-007）
- 新认证 provider 必须继承 BaseAuthProvider

## 建议
- 遵循 test_authenticator.py 中的测试模式

## 否定性发现（不存在的东西）
- 没有现成的 CSV 解析工具（需要新增依赖）
- 没有统一的分页 helper（各端点自己实现）
```

❌ 差的调研：
```
看了一下代码，用 JWT 认证，挺简单的
```

## 调研驱动的问题清单

开始调研前先列问题，调研完就结束：
1. 数据库迁移是怎么处理的？
2. 是否有现成的日期格式化工具函数？
3. 认证中间件的模式是什么？
4. API 错误响应的约定是什么？
5. 环境配置是如何管理的？

## 调研如何影响实现范围

好的调研经常会：
- 发现已有工具，把范围减少 80%
- 发现架构约束，在写了需要丢弃的代码前调整方向
- 把"我以为需要从头写"变成"只需要配置"

## 反面模式

❌ 边写边调研：在生成代码的同时"即时调研"
- 感觉高效，实际信息不完整就做了决策
- 结果：隔离能跑，集成烂掉

❌ 调研摘要和实现用同一个 Agent：
- 调研产出的 context 污染实现阶段
- 正确做法：调研结束 → 写发现文档 → 新 Agent 只读发现文档
