# 因素 III：一个 Agent 一个任务

**规则：每个 Agent 获得一个范围明确的任务和全新的 context。永远不要复用已饱和的窗口。**

## 核心逻辑

- 研究完认证系统的 Agent，是实施修复的最差人选——窗口里满是研究上下文
- 不同阶段需要不同 context，不该把它们混在一起

## 何时重新生成一个 Agent

✅ 应该重新生成：
- 阶段转换（调研→规划→实现→验证）
- context 窗口 >70% 已满
- 完成一个 Issue 准备开始下一个不相关的 Issue
- Agent 开始自相矛盾或引用过时信息
- 任务范围发生显著变化
- 已经用同一方法重试 3 次以上没有进展

❌ 不需要重新生成：
- 调试中途的紧密反馈循环
- 需要快速澄清的来回沟通
- 小幅范围调整
- 测试失败后的直接修复

## 交接协议

阶段完成时的标准步骤：
1. 产出工件（发现/计划/代码/结果），提交到 Git
2. 写提炼摘要：引用具体文件路径，列出下一阶段依赖
3. 生成新 Agent，只加载提炼后的工件

**交接文档格式：**
```
HANDOFF:
- Goal: [目标]
- Done: [完成了什么]
- Decisions: [关键决策]
- Next: [下一步]
- Key files: [相关文件路径]
```

## 一个 Issue 一个 Agent

❌ 错误：给一个 Agent 分配多个 Issue
- 认知负担：同时跟踪多个问题域
- 上下文交叉污染
- 故障影响范围扩大

✅ 正确：
- 一个 Agent，一个 Issue，全神贯注
- 多个 Issue → 并行多个 Agent
- 编排层（人类/协调 Agent）管理批量，工作 Agent 处理原子单元

## 饱和检测信号

以下任一出现，说明 Agent 已饱和：
- 重复说过的话
- 忘记已经尝试过的方法
- 提议与自己之前的建议矛盾
- 询问已经给过的信息
- 修改之前不该动的文件
