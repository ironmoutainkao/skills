# 因素 V：外部验证

**规则：任何 Agent 都不得给自己的工作打分。永远如此。**

## 为什么自我验证无效

自我验证 = 用生成时完全相同的假设和盲区来验证工作。
就像作者自己校对自己的手稿，读到的是自己"打算写的"，而不是实际写的。

**自我验证的核心问题：**
- 编写代码的 Agent 知道代码"应该"做什么 → 用意图匹配实现，而不是客观审查
- 修复 bug 的 Agent 知道修复为什么应该有效 → 只测试自己设计的正常路径

## 验证层级（从弱到强）

1. 同一 Agent 同一会话中稍后 → 几乎没用，上下文相同
2. 同一 Agent 不同会话 → 稍好一些，低风险工作可接受
3. 不同 Agent 同一模型 → 良好，不同 context 不同生成路径
4. 不同模型 → 更好，不同训练不同偏见
5. 多模型委员会（3+ 模型） → 最适合主观决策（架构/设计/优先级）
6. 自动化测试套件 → 最适合客观正确性（不在乎意图，只在乎行为）
7. 人类专家审核 → 最适合高风险决策（安全/法律/架构）

## 什么必须人类验证

- **安全决策**：实现由 Agent，验证必须有人
- **架构决策**：涉及长期目标和团队能力
- **法律合规**：责任是真实的
- **不可逆决策**：数据库迁移、数据删除、API 废弃
- **面向用户的决策**：产品方向、定价、UX 变更

## 标准验证流程示例

| 工作类型 | 执行者 | 验证者 | 通过条件 |
|---------|--------|--------|---------|
| 代码变更 | Agent A | Agent B（不同会话） | B 批准 PR 或提问 |
| 测试 | 写代码的 Agent | 测试套件 | 所有测试通过 |
| 架构设计 | Agent 或人类 | 多模型委员会 | 对方案共识 |
| 安全实现 | Agent | 安全模型 + 人类工程师 | 两者都批准 |
| 生产部署 | Agent | 测试套件 + 人工运维 | 测试通过 + 人工签字 |

## 测试驱动开发是最纯粹的外部验证

TDD 先写测试（规格），再写代码（实现）——测试的上下文与实现的上下文是不同的。

更好：测试由不同于编写代码的 Agent 来写。

## 快速检查

- [ ] 是否有 Agent 在自己验证自己的工作？
- [ ] 验证强度是否和决策风险匹配？
- [ ] 安全/架构/不可逆操作有没有人工把关？
