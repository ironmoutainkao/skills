# 因素 X–XII：规模层级（多 Agent 工作流）

> 这三个因素属于高级模式，**不是单 Agent 工作流的必要条件**。
> 只有当你并行运行多个 Agent 时，它们才变得关键。

---

## 因素 X：隔离工作单元

**规则：每个 Worker 拥有自己的工作空间、自己的 context，以及零共享可变状态。**

### 真正的隔离意味着

- 每个 Worker 使用**独立的 git worktree**（不只是分支）
- 每个 Worker 有**全新的 context**（不共享对话历史）
- **无渗透**：一个 Worker 的变更对其他 Worker 不可见，直到合并
- **独立故障域**：一个 Worker 崩溃，其他继续

### 为什么需要隔离

Agent A 修改了 `config.yaml` 未提交 → Agent B 读取，误以为是已提交的内容 → 基于污染的状态做决策 → 两个 Agent 都没意识到，仓库进入不一致状态。

这不是假设，这是共享工作目录时必然发生的。

### 隔离实现

```bash
# 每个 Worker 独立的 worktree
git worktree add ../workers/worker-a -b feature/auth-refactor
git worktree add ../workers/worker-b -b feature/db-migration

# 失败时：删除 worktree，生成新的
rm -rf workers/worker-failed
git worktree add workers/worker-retry -b feature/retry
```

## 因素 XI：层级化监督

**规则：升级流向上传递，绝不横向流转。**

### 监督树结构

```
人类（根）
  ↓
协调 Agent（Mayor/监督者）
  ↓
子任务监督（Witness/Refinery）
  ↓
工作 Agent（Worker/Polecat）
```

### 核心原则

- 工作 Agent：本地重试 3 次，不行就**向上升级**，不横向寻求帮助
- 监督者：决定重启/重新分配/重新定义/继续升级
- 人类在**顶层**，不是嵌在每一次重试循环里

### 失败处理模式

✅ 正确：Agent 失败 → 重试（有界）→ 升级给监督者（带结构化报告）→ 监督者决策
❌ 错误：Agent 失败 → 求助人类 → 人类修复 → Agent 继续 → 下一个失败 → 再求助（人类成为瓶颈）

### 扁平拓扑的问题

- 协商导致死锁（A 等 B，B 等 A，没人能打破僵局）
- 无限重试（没人能决定"够了，升级"）
- 缺乏全局视角（每个 Agent 只看到自己的 context）

## 因素 XII：从失败中收获智慧

**规则：失败的尝试就是数据，用与成功同等的严谨度提取和索引它们。**

### 为什么失败比成功更有价值

失败 = 负面知识 = 修剪搜索空间。
知道"当 Y 为真时不要尝试 X"，比知道"用 Z 解决这个问题"更能防止下一次浪费。

### 失败捕获格式

```yaml
failed_attempt:
  id: "fa-20260215-001"
  task: "解析自定义配置格式"
  approach: "正则表达式匹配"
  context:
    - "配置看起来有规则结构"
  failure_mode: "在第 45-67 行的嵌套结构上失败"
  hypothesis: "自定义格式支持嵌套，但前 40 行看不出来"
  next_approach: "尝试 YAML 解析器"
  timestamp: "2026-02-15T14:23:01Z"
```

### 失败转化为文档

```markdown
## 配置解析：不要做什么

- **正则表达式用于嵌套结构**（n=12 次失败）
  - 失败条件：嵌套深度 > 2，或前段扫描看不出嵌套
  - 原因：正则无法处理递归结构
  - 替代方案：使用 JSON 或自定义解析器

- **标准 YAML 解析器用于自定义标签**（n=8 次失败）
  - 失败条件：供应商使用 YAML 扩展语法
  - 替代方案：预处理去掉扩展，或用供应商专用解析器
```

### 失败率趋势是健康指标

- 第 1 个月：40% 尝试失败 → 第 6 个月：15% → 第 12 个月：10%（只有新颖失败）
- 如果失败率不下降：你在重复支付同样的学费
- 零重复失败 = 完美的失败收获
