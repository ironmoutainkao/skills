# 因素 I：上下文即一切

**原则：管理进入 context window 的内容，就像管理进入生产环境的内容一样。**

## 核心问题

- LLM 对 context window 的开头和结尾注意力强，中间容易丢失（"迷失在中间"效应）
- context 越长，质量越差，而不是越好
- 每个加载的文件、搜索结果、日志都是一个决策

## context 饱和的症状

1. Agent 反复重复自己说过的话
2. Agent 忽略最近的指令（被噪声淹没）
3. Agent 产生幻觉（在噪声上模式匹配）
4. Agent 自相矛盾（遗忘了之前的方案）
5. Agent 请求已经提供过的信息

## 良好实践

**1. 基于阶段的 context 窗口**
- 调研阶段：加载探索所需片段，结束时写摘要
- 规划阶段：加载调研摘要（不是完整对话记录）
- 实现阶段：加载计划文档 + 正在编辑的文件
- 验证阶段：加载实现摘要 + 测试

**2. 主动压缩（不是自动压缩）**
- 阶段切换时，自己写交接摘要：目标/完成了什么/决策/下一步
- 1 万 token 的对话 → 200 token 的核心摘要
- 绝不直接传递完整对话历史

**3. 激进摘要化**
- 每个子任务结束后：3 行摘要
- 每个会话结束后：写交接说明
- 每个阶段结束后：压缩成结构化文档

**4. 先搜索再加载**
- 不要告诉 Agent "读取整个 src/ 目录"
- 用 grep 找相关函数，用 AST 工具只提取类定义
- 加载回答当前问题所需的最小 context 切片

**5. 一次性工作用草稿会话**
- 快速问答开新会话，问完关掉
- 不要用一次性工作污染主会话 context

## 反面案例

❌ 8 万 token 的调试会话（加载了大量无关代码）→ Agent 游荡 5 个回复找不到 bug
✅ 9K token 精准上下文（错误日志 + 相关函数 + 对应测试）→ 1 个回复找到 bug

## 快速检查清单

- [ ] 是否知道 context 里加载了什么，以及为什么？
- [ ] 阶段切换时是否写了摘要而不是直接传历史？
- [ ] 是否有一次性工作在污染主会话？
- [ ] Agent 是否开始重复自己或忽略指令？（说明已饱和）
