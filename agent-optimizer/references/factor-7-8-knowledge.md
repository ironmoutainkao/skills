# 因素 VII & VIII：经验提取 + 知识复利

## 因素 VII：提取经验教训

**规则：每次会话产出两个成果——工作产物 + 经验教训。**

### 为什么要提取

不提取 = 组织失忆。会话打开时的知识，标签关掉就消失。

**缺少提取的症状：**
- 相同 bug 跨会话重复出现
- 不断重复解释（新人总在问相同的问题）
- 害怕重构（没人知道为什么是这个结构）

### 什么值得提取

✅ 高价值：
- 决策及其理由（"我们选 X 不选 Y，因为 Z"）
- 非显而易见的模式（"这个模块用工厂模式处理 N 个边缘情况"）
- 失败模式（"尝试了依赖注入，循环导入，已回退"）
- 集成陷阱（"API 要求 header 和查询参数都传 auth token"）

❌ 低价值：
- 从代码可以推导出的显而易见的事实
- 逐步记录做了什么（那是提交信息的事）
- 模糊的印象没有具体细节

**检验标准：** 这能帮助某人（包括未来的你）避免浪费时间或做出更好决策吗？

### 好的提取格式：决策记录

```markdown
# 决策：使用 SQLite 作为本地缓存（而非 Redis）

## 背景
需要为 API 响应提供持久化缓存。

## 决策
使用 SQLite，TTL 5 分钟。

## 理由
- 无外部依赖（Redis 需要单独进程）
- 对 <10k 条目性能足够
- 基于文件：易于备份/调试

## 考虑过的替代方案
- Redis：性能更好但运维复杂度高
- 内存字典：重启后无持久化

## 后续触发条件
- 如果缓存增长超过 10k 条目，重新评估 Redis

## 标签
#architecture #caching #sqlite
```

### 好的提取格式：失败文档

```markdown
# 失败方法：基于 Token 的 WebSocket 认证

## 尝试了什么
在 WebSocket URL 查询参数中传递 JWT。

## 为什么失败
- 查询参数以明文记录在日志中（安全问题）
- URL 长度限制，token 可能超限
- 浏览器历史会捕获完整 URL

## 改用了什么
使用 Sec-WebSocket-Protocol header 传递 token（RFC 6455）。

## 教训
永远不要在 URL 中放密钥，即使使用 HTTPS。

## 标签
#websocket #authentication #failures #security
```

## 因素 VIII：知识复利

**规则：学习成果必须自动回流到未来的会话中。**

### 飞轮模式

```
提取 → 质量过滤 → 存储 → 注入 → 引用 → 衰减（修剪过期知识）
           ↑_________________________________________|
```

**复利方程：`retrieval_rate × citation_rate > decay_rate`**

如果不等式不成立，知识会衰减到零。

### 四个关键步骤

**提取**：会话结束时结构化捕获（不是可选步骤，不是有时候做）

**过滤**：进知识库前把关
- 是否可操作？
- 是否具体？
- 是否已验证（测试过，不是直觉）？
- 是否唯一（不是已有内容的重复）？

**注入**：会话开始时，在 Agent 写第一行代码前，注入相关知识
- "上次修改这个模块时学到的是..."
- "我们不用库 X 的原因是..."

**衰减**：定期修剪 N 次会话内未被引用的知识
- 过时的知识比没有知识更糟糕（它会误导）
- 代码在变，API 在演进，旧建议会变成错误

### 只写数据库问题

大多数团队有提取没有注入，知识躺在 `/docs` 里无人问津。
Agent 不会主动去搜索 47 份回顾报告。注入必须是**主动的、自动的**。

### 可观察的复利效果

❌ 没有复利：
- Agent 建议已废弃的库
- 每次遇到支付 API 都重新调试限流

✅ 有复利：
- Agent 开场就说"我看到之前尝试了 X 因为 Y 失败了，我会用 Z"
- 第 50 次会话比第 1 次会话用更少的时间、更少的错误完成类似任务
